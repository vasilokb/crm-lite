# Спецификация MVP: Локальная CRM-lite для выставочного агентства  
**Редакция 7.0 — основной документ, описывающий MVP**

> Этот документ является основным и приоритетным описанием MVP.  
> При разработке и ревью ориентироваться на него, предыдущие черновики (`mvp.md`) считать устаревшими.

---

## 1. Бизнес-контекст и фокус MVP

**Задача продукта:**  
Помочь менеджерам агентства выставочных стендов не терять входящие запросы, вести их по воронке и контролировать связанные контакты, компании, сделки и задачи, чтобы понимать:

- сколько потенциальной выручки сейчас в работе;
- какие сделки “застряли”;
- какие задачи просрочены.

**Ядро MVP (минимально жизнеспособный функционал):**

- Списки и карточки лида, компании, контакта и сделки.
- Конвертация лида в рабочие сущности.
- Воронка сделок и базовые правила переходов (`won`/`lost`).
- Активности (notes/tasks) на уровне сделки.
- Отдельный Dashboard с минимальным набором KPI и диаграмм.

Расширенные функции (сложная аналитика, роли пользователей, кастомизация интерфейса и т.п.) **не входят в MVP**, сознательно исключены.

---

## 2. Технологический стек и ограничения

- **Фронтенд:** Next.js, React, TypeScript.
- **База:** PostgreSQL.
- **ORM:** Prisma (`prisma@6.19.3`, `@prisma/client@6.19.3`), без `@prisma/adapter-pg` и `prisma.config.ts`.
- **Валидация:** Zod на серверной стороне.
- **Аналитика / графики:** `chart.js@4.5.1` и `react-chartjs-2@5.3.1`.
- **Режим работы:** локальная CRM-lite, без realtime, websockets и push-обновлений.

## 3. Слой данных: модели и контрольные данные

### 3.1. Модели Prisma

Обязательные сущности:

- **Lead**
  - Входящий запрос.
  - Поля: идентификатор, имя/название, обязательное поле `source` со значениями `site`, `email`, `phone`, `referral`, `manual`; статус (`new`, `processed`, `converted` и т.п.), бюджет, сроки, комментарий, ответственный.
  - После конвертации может быть связан с `Account`, `Contact` и `Opportunity`.

- **Account**
  - Компания-заказчик.
  - Связи: несколько `Contact`, несколько `Opportunity`.

- **Contact**
  - Контактное лицо.
  - Связан с `Account` (Many-to-One).
  - Может участвовать в нескольких `Opportunity`.

- **Stage**
  - Справочник стадий воронки продаж (например, `qualification`, `proposal`, `negotiation`, `won`, `lost`).
  - Используется как ссылка в сделках (`stageId`).

- **Opportunity**
  - Сделка/потенциальный проект.
  - Поля: идентификатор, название, сумма (`amount`), статус (`open`, `won`, `lost`), внешний ключ `leadId` (исходный лид), `stageId` (Stage), связи с `Account` и `Contact`.

- **Activity**
  - Активности, привязанные к сущностям (в MVP в первую очередь к `Opportunity`).
  - Поля: тип (`note` или `task`), текст, дата создания.
  - Для `task`: обязательные поля `dueDate` и `done` (boolean).

### 3.2. Seed и контрольные данные

Seed-скрипт детерминированно создаёт:

- 6 лидов с разными `source` и статусами.
- 4 компании (Accounts).
- 5 контактов, связанных с компаниями и сделками.
- 6 сделок (Opportunities) с разным распределением по стадиям.
- 8 активностей:
  - среди задач: минимум 2 просроченные (`dueDate < today`) и минимум 2 на сегодня.

Эти данные используются для прохождения контрольного сценария через UI без прямого доступа к базе.

## 4. Интерфейс: списки и навигация (Drawer-ориентированный подход)

### 4.1. Общая модель интерфейса

Интерфейс строится вокруг трёх уровней:

1. **Списки сущностей** (Leads, Accounts, Contacts, Opportunities) — табличный режим.
2. **Drawer-карточки** — боковые шторки для просмотра/редактирования и работы с связями.
3. **Отдельный Dashboard** — экран агрегированных показателей.

### 4.2. Табличные списки

Для разделов (лиды, компании, контакты, сделки):

- Табличный вид до 50 записей на страницу с пагинацией.
- Строка поиска above the fold:
  - работает по имени/названию, компании, названию сделки.
- Фильтры:
  - Лиды:
    - по `source` (`site`, `email`, `phone`, `referral`, `manual`);
    - по статусу лида.
  - Сделки:
    - по стадии (`Stage`);
    - по статусу (`open`, `won`, `lost`).

### 4.3. Цветовое кодирование в списках

Цветовые бейджи используются **точечно**:

- Статусы:
  - `won` — зелёный бейдж.
  - `lost` — красный бейдж.
  - `open` — нейтральный/синий.
- Просроченные задачи в специальных списках — отдельная подсветка.
- Остальные поля — нейтральные (без тотальной раскраски таблицы).

### 4.4. Отображение множественных связей в таблицах

Для связей One-to-Many (например, у компании несколько контактов и сделок):

- В ячейке отображается **кликабельный бейдж-счётчик**:
  - `Contacts: 3`, `Deals: 2`.
- По клику:
  - открывается Drawer компании;
  - фокус смещается на блок “Связанные списки” с полным перечнем контактов/сделок.

В таблицах не выводятся длинные списки имён/названий — это делается в Drawer.

## 5. Drawer-карточки: структура, связи и ошибки

### 5.1. Общий layout Drawer и ошибки

Каждый Drawer имеет фиксированную структуру:

1. **Шапка**
   - Название сущности.
   - Текущий статус (бейдж).
   - Краткие ссылки на родительские сущности (например, компания из контакта).
   - Для сделок — визуальный прогресс-бар стадий.

2. **Блок “Данные” (форма)**
   - Основные редактируемые поля сущности.
   - Серверная валидация через Zod.
   - Ошибки **валидации/бизнес-логики**:
     - подсветка проблемных полей;
     - локальные тексты рядом с полями;
     - общий текст в зоне формы (“Исправьте выделенные поля…”).
   - При падении сервера (ошибка 500 или аналогичная):
     - форма не зависает в вечном `Loading`;
     - внизу шторки или в виде глобального тоста выводится сообщение:  
       **“Не удалось сохранить изменения, попробуйте позже”**.

3. **Блок множественных связей (Related Lists / Tabs)**
   - Область ниже формы с собственной вертикальной прокруткой.
   - Показывает One-to-Many связи (контакты, сделки, активности).

4. **Блок активностей / быстрых действий (для сделок)**
   - Лента (timeline) `Activity`.
   - Быстрые действия по задачам.

### 5.2. Вложенная навигация внутри Drawer

В блоках “Связанные сущности” и “Активности” элементы (компания, контакт, сделка) кликабельны.

**Поведение:**

- В любой момент существует **один активный Drawer**.
- Если пользователь находится в Drawer (например, компании) и кликает на связанный контакт/сделку:
  - новая шторка поверх не открывается;
  - содержимое текущего Drawer **заменяется** карточкой выбранной сущности (Contact/Opportunity);
  - URL обновляется (например, `/accounts/:id` → `/contacts/:id`).
- Это избавляет от “модального ада” и обеспечивает предсказуемое поведение кнопки “Назад” браузера.

## 6. Конвертация лида (Convert Lead)

Конвертация — ключевой сценарий, выполняется из Drawer лида.

### 6.1. UX-логика и физика формы

- В шапке лида — кнопка `Convert lead`.
- При нажатии `Convert lead` **внутри того же Drawer**:
  - плавно раскрывается **inline-форма по паттерну Accordion**;
  - форма:
    - показывает текст-подтверждение (“Будут созданы: Компания, Контакт, Сделка (опционально)”);
    - выводит недостающие обязательные поля для конвертации (например, имя контакта, название компании, ключевые поля сделки).
  - Конвертация выполняется **без перехода на новый экран или модальное окно**: менеджер работает в том же контексте.

- Конвертация **не проходит** без обязательных данных:
  - поля, мешающие конвертации, подсвечиваются;
  - рядом выводится пояснение (например, “Укажите имя контакта для конвертации лида”).

- После успешной конвертации:
  - статус лида меняется на “converted/обработан”;
  - действие `Convert lead` блокируется (кнопка отключена или заменена текстом “Лид уже конвертирован”);
  - в карточке лида появляются кликабельные ссылки на созданные `Account`, `Contact`, `Opportunity`.

### 6.2. Бизнес-правила и атомарность конвертации

- Convert lead создаёт связанные:
  - `Account`,
  - `Contact`,
  - при необходимости — `Opportunity`.
- Один и тот же лид нельзя конвертировать дважды.
- После конвертации связи корректно видны:
  - в карточке лида;
  - в карточке компании;
  - в карточке контакта;
  - в карточке сделки.

**Требование атомарности:**

- Конвертация выполняется **строго в рамках одной транзакции базы данных**:
  - использовать Prisma `$transaction` для создания `Account`, `Contact`, `Opportunity` и изменения статуса `Lead`.
- В случае сбоя на любом этапе:
  - вся операция **полностью откатывается**;
  - не должно возникать “битых” данных (например, созданная компания без контакта).
- Пользователь в UI:
  - видит сообщение об ошибке конвертации (в зоне Accordion);
  - данные лида остаются в исходном состоянии.

## 7. Воронка сделок, стадии и активности

### 7.1. Визуальная воронка сделок

- В таблице сделок:
  - каждая сделка имеет бейдж стадии (Stage).
- В Drawer сделки:
  - стадия отображается как **прогресс-бар** (цепочка стадий);
  - клик по другой стадии инициирует попытку смены.

### 7.2. Согласованность стадии

- Стадия в списке и в Drawer **всегда совпадает**.
- Изменение стадии:
  - инициируется из прогресс-бара;
  - после успешного сохранения обновляется и в списке, и в Drawer.

### 7.3. Жёсткие правила переходов (`won`/`lost`)

- Переход в `won` запрещён, если:
  - не заполнена сумма сделки (`amount`);
  - нет связанного контакта.
- Переход в `lost` запрещён, если:
  - не указана текстовая причина отказа (`reasonLost` или аналогичное поле).

При нарушении:

- форма остаётся открытой;
- проблемные поля подсвечиваются;
- выводится явное сообщение о необходимости заполнения.

### 7.4. Активности (Activity) и задачи (Optimistic UI)

- Типы:
  - `note` — текстовая заметка.
  - `task` — задача с `dueDate` и `done`.

- В Drawer сделки:
  - отображается timeline активностей:
    - `note` и `task` визуально различаются (иконки/цвет);
    - просроченные задачи дополнительно выделяются.

- Для каждой задачи:
  - есть чекбокс/кнопка `Завершить`, меняющая `done`.

**Паттерн Optimistic UI:**

- При клике по чекбоксу задачи:
  - статус `done` **мгновенно меняется в UI**;
  - запрос к серверу отправляется в фоне;
  - при успехе состояние подтверждается;
  - при ошибке:
    - чекбокс возвращается в исходное состояние;
    - показывается короткое сообщение “Не удалось завершить задачу, попробуйте позже”.

- Форма добавления активности:
  - выбор типа (`note`/`task`);
  - текст;
  - для `task` — обязательный `dueDate`;
  - кнопка `Add` с валидацией и отображением ошибок по общим правилам.

## 8. Dashboard: отдельный экран

Dashboard — отдельный маршрут, не часть карточек и не flowchart.

### 8.1. KPI-карточки

Above the fold видны четыре ключевых показателя:

- общее количество лидов;
- количество открытых сделок (`open`);
- сумма открытых сделок;
- количество просроченных задач.

### 8.2. Диаграммы Chart.js

- Диаграмма по сделкам в разрезе стадий (`Stage`).
- Диаграмма по статусам лидов.

Архитектурный принцип:

- все агрегации выполняются на бэкенде через Prisma `groupBy` / `aggregate`;
- компоненты диаграмм на фронте **“тупые”**:
  - получают через props готовые массивы `labels: string[]` и `values: number[]`;
  - не содержат логики выборки/агрегации.

### 8.3. Операционные списки

- **Recent Leads** — список последних созданных лидов.
- **Overdue Tasks** — список задач:
  - `dueDate < today`;
  - `done = false`.
- Опционально: **Stuck Deals** (сделки в одной стадии дольше определённого срока), если не перегружает экран.

### 8.4. Обновление Dashboard

- После создания/редактирования лида или сделки:
  - при повторном переходе на Dashboard делается свежий HTTP-запрос;
  - realtime и websockets не используются.

## 9. Стратегия удаления, ограничений и целостности

### 9.1. Удаление и ограничения (Constraints)

В MVP логика удаления сущностей **жёстко ограничивается**, чтобы не ломать демонстрационный сценарий:

- Для связей вроде:
  - `Account` → `Contact`,
  - `Account` → `Opportunity`,
  - `Opportunity` → `Activity`,
- рекомендуется настроить `onDelete: Restrict` или эквивалентный механизм в Prisma schema:
  - сущность с зависимыми данными **не может быть удалена** через обычные операции.

**Интерфейсный принцип:**

- В Drawer-форме **нет кнопки “Удалить”** для:
  - компании;
  - контакта;
  - сделки;
  - активностей.
- Любые деструктивные операции (архивирование, soft delete) в MVP не реализуются, статусами/переходами (`won`/`lost`) и завершением задач покрываются основные потребности.

Это защищает контрольный сценарий от случайной поломки и возникновения несогласованных данных.

## 10. Локальный запуск и packaging к сдаче

MVP должен быть:

- полностью запускаемым локально по `README` (без правки кода под окружение);
- с `.env.example` (без секретов);
- с Prisma schema и миграциями или описанным воспроизводимым dev-путём применения схемы;
- с командами `seed`/`reset` для контрольных данных.

В проектном пакете:

- исходный код CRM-lite;
- `README.md` с:
  - локальным запуском,
  - seed/reset,
  - правилами безопасного изменения схемы,
  - описанием демонстрационного сценария:
    - запуск → seed/reset → лид → convert lead → сделка → активность → воронка → dashboard;
- краткий список известных ограничений (что сознательно не сделано в MVP: нет ролей, нет мобильной версии, нет удаления сущностей).
